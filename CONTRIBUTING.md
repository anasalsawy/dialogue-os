# Contributing to Dialogue OS

Contributions are welcome, but Dialogue OS is a governed doctrine project. Contributions must strengthen clarity, safety, accountability, implementation realism, and constitutional consistency.

## Contribution principles

Good contributions should:

- Improve the doctrine without weakening human sovereignty.
- Make agent responsibilities clearer.
- Strengthen auditability, evidence, escalation, and memory discipline.
- Improve implementation realism.
- Remove ambiguity.
- Make the system safer and more accountable.
- Preserve the public/private boundary.

Bad contributions include:

- Hype without operational value.
- Roleplay language that weakens accountability.
- Claims that the system already works without evidence.
- Private credentials, secrets, logs, account data, payment data, or customer data.
- Unsafe automation workflows.
- Unreviewed implementation code pretending to be doctrine.
- Additions that remove human control or make agents unaccountable.

## Contribution license

By opening an issue, pull request, or contribution, you agree to the contribution terms in [`OWNERSHIP.md`](OWNERSHIP.md).

In plain language:

- You keep copyright over your contribution.
- You grant the project and Anas Alsawy broad rights to use and adapt it.
- You do not gain ownership over Dialogue OS.
- You do not gain rights to the private runtime.
- You may not claim to be the official owner, maintainer, or commercial provider unless authorized.

## Pull request checklist

Before submitting a pull request, confirm:

- [ ] The change contains no secrets, credentials, private account data, customer data, or payment information.
- [ ] The change does not include unsafe or illegal operational instructions.
- [ ] The change strengthens the doctrine instead of making it vague.
- [ ] The change respects the authority chain: Human > Codex/Chief > Leads > Workers.
- [ ] The change preserves Watcher independence.
- [ ] The change uses evidence-focused language.
- [ ] The change does not claim runtime completion without proof.

## Preferred contribution areas

- Governance language improvements.
- Watcher scoring refinements.
- Agent role definitions.
- Memory cycle design.
- War Room architecture.
- Audit and evidence standards.
- Public roadmap refinement.
- Diagrams and explanatory documentation.

## Maintainer review

The maintainer may reject, edit, squash, restructure, or delay contributions for any reason, especially if they affect project identity, safety, public positioning, or ownership clarity.
